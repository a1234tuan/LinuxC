## 二、章节级知识提炼（示例：前3章）

### **第1章 软件与软件工程概述**

#### 1️⃣ 核心知识点

- **软件的概念与特性**
  - 软件 = 程序 + 数据 + 文档
  - 特性：无形性、智力性、设计性、可维护性、可复制性
- **软件危机**
  - 原因：复杂性、缺乏管理、需求不确定、质量难控
  - 表现：超预算、延期、质量差、维护困难
- **软件工程**
  - 定义（IEEE）：以系统化、规范化、可度量的方法开发和维护软件
  - 目标：高质量、低成本、可维护、按时交付
- **生命周期阶段**
  - 定义 → 开发 → 运行与维护
- **工程方法与工具**
  - 三要素：过程、方法、工具
  - 方法学分类：结构化、面向对象、面向服务、面向数据、形式化方法

#### 2️⃣ 前置依赖与关联

- 前导：计算机科学基础
- 输出成果（如需求规格说明书）将成为系统设计阶段的输入。

#### 3️⃣ 学习目标

- 能解释“软件工程化”的意义与目标
- 能描述软件生命周期的主要阶段
- 能区分结构化、面向对象与面向服务方法

#### 4️⃣ 推荐学习路径

1. 阅读PPT并理解“软件危机”案例
2. 观看案例视频（推荐关键词：“软件危机 历史 回顾”）
3. 比较不同开发模型的工程特点
4. 小型项目：描述一个简单系统的生命周期阶段

#### 5️⃣ 达成标准与自测

- **掌握标准**：能用自己的话定义“软件工程”并解释工程化思想。
- **自测题**：
  1. 为什么说软件是“设计出来的”而不是“制造出来的”？
  2. 举例说明软件危机在现代软件开发中的表现。

#### 6️⃣ 推荐资源

- 《软件工程（第9版）》Sommerville 第1章
- B站搜索：“软件工程概述 软件危机案例分析”

### **第2章 软件生存期模型**

#### 1️⃣ 核心知识点

- **软件过程与模型**
  - 软件过程：完成软件开发的一系列活动框架
  - 软件过程模型：瀑布、快速原型、增量、螺旋、喷泉、统一过程、CBSE、敏捷
- **典型模型特征**
  - **瀑布模型**：顺序、文档驱动、需求稳定时适用
  - **快速原型**：快速验证需求，适合探索性项目
  - **增量模型**：分阶段交付，可逐步完善
  - **螺旋模型**：风险驱动、强调迭代与原型
  - **统一过程（RUP）**：用例驱动、以架构为核心
  - **敏捷过程**：强调迭代交付、客户协作、响应变化

#### 2️⃣ 前置依赖与关联

- 承接第1章“生命周期与方法论”内容
- 各模型的产出（需求文档、设计文档等）是后续阶段的输入。

#### 3️⃣ 学习目标

- 能区分主流过程模型的结构与适用场景
- 能针对项目特征选择合适的开发模型

#### 4️⃣ 推荐学习路径

1. 阅读各模型定义与流程图
2. 比较瀑布、敏捷与螺旋模型的优缺点
3. 案例分析：以“图书管理系统”为例比较三种模型开发差异
4. 小型实践：绘制RUP二维模型结构

#### 5️⃣ 达成标准与自测

- **掌握标准**：能判断项目应采用哪种开发模型并说明理由。
- **自测题**：

  2. 画出螺旋模型四象限活动，并说明每个阶段的核心任务。

#### 6️⃣ 推荐资源

- 《Sommerville》第4章 软件过程
- B站：“敏捷开发实战案例”“RUP生命周期详解”

### **第3章 需求工程与结构化分析**

#### 1️⃣ 核心知识点

- **需求工程概念**
  - 从沟通 → 获取 → 分析 → 规格说明 → 确认 → 管理
  - 软件需求类型：功能性需求、非功能性需求、约束条件
- **需求工程任务**
  - 获取（Elicitation）
  - 分析（Analysis）
  - 定义（Specification）
  - 验证（Validation）
- **需求获取方法**
  - 访谈、问卷、观察、原型、用例分析
- **结构化分析方法**
  - 数据流图（DFD）、数据字典、状态图
  - 自顶向下、逐层分解思想
- **需求规格说明（SRS）**
  - 内容：功能描述、性能需求、接口说明、约束条件等
  - 质量要求：正确、无歧义、完整、一致、可验证

#### 2️⃣ 前置依赖与关联

- 来源：软件生命周期“定义阶段”
- 输出：SRS文档，为系统设计提供输入。

#### 3️⃣ 学习目标

- 能描述需求工程各阶段的任务与目标
- 能使用DFD图描述系统功能需求
- 能编写简化版SRS

#### 4️⃣ 推荐学习路径

1. 阅读需求工程章节并识别七个动作（起始→管理）
2. 使用案例法练习需求获取（如“图书借阅系统”）
3. 使用Visio或PowerDesigner绘制DFD与ER图
4. 完成一次需求评审与版本变更记录

#### 5️⃣ 达成标准与自测

- **掌握标准**：能输出完整的SRS，逻辑清晰、无冲突。
- **自测题**：
  1. 请解释“验证（Verification）”与“确认（Validation）”的区别。
  2. 使用数据流图描述学生成绩管理系统的主要功能。

#### 6️⃣ 推荐资源

- 《Sommerville》第6章 需求工程
- 工具实践：PowerDesigner / Visio
- B站：“软件需求分析实战课程”

## **第4章 软件设计**

### 1️⃣ 核心知识点

#### （一）软件设计的概念

- 软件设计：**将需求转化为可实现的软件系统蓝图**的过程。
- 两个阶段：
  - **概要设计（系统结构设计）**：确定模块划分与接口关系；
  - **详细设计**：确定每个模块的算法与数据结构。

#### （二）设计原则

- **抽象（Abstraction）**：从复杂中提取本质。
- **逐步求精（Stepwise Refinement）**：逐步细化设计。
- **模块化（Modularity）**：系统由相互独立、协作的模块组成。
- **信息隐藏（Information Hiding）**：模块内部实现细节对外部隐藏。
- **高内聚、低耦合（Cohesion & Coupling）**：衡量模块质量的关键指标。

#### （三）结构化设计

- 基于数据流分析结果的**自顶向下**设计方法。
- 常用图形工具：
  - **结构图（Structure Chart）**
  - **模块层次关系图**
- 设计策略：
  - 转换分析（Transform Analysis）
  - 事务分析（Transaction Analysis）

#### （四）面向对象设计（OOD）

- 核心思想：**以对象为中心**，统一数据与行为。
- 三个主要模型：
  - **对象模型（Object Model）**
  - **动态模型（Dynamic Model）**
  - **功能模型（Functional Model）**
- UML主要图形：
  - 类图、顺序图、活动图、状态图、用例图、组件图。

#### （五）体系结构设计

- 描述系统高层结构的“骨架”。
- 常见体系结构风格：
  - 分层式、客户机-服务器、管道-过滤器、MVC、微服务。
- **接口设计**：定义模块之间通信的输入输出与调用关系。

## **第5章 软件实现与测试**

### 1️⃣ 核心知识点

#### （一）软件实现

- 实现阶段任务：将设计文档转换为可执行程序。
- 编码规范：
  - 命名一致、缩进规范、注释完整；
  - 代码可读性与可维护性优先；
  - 使用版本控制（如Git）。
- 常见问题：代码冗余、接口不一致、未遵循设计规范。

#### （二）软件测试

- 定义：在规定条件下执行程序，发现错误并评估质量的过程。
- **测试目的**：验证（Verification）与确认（Validation）。
  - 验证：“我们是否正确地构建了产品？”
  - 确认：“我们是否构建了正确的产品？”

#### （三）测试分类

| 层级     | 主要目标           | 示例                |
| -------- | ------------------ | ------------------- |
| 单元测试 | 验证模块正确性     | JUnit、CppUnit      |
| 集成测试 | 验证模块接口       | Selenium、WinRunner |
| 系统测试 | 验证整个系统功能   | LoadRunner          |
| 验收测试 | 用户确认需求满足度 | Beta测试            |

#### （四）测试方法

- **黑盒测试**：基于输入输出，关注功能。
  - 方法：等价类划分、边界值分析、判定表、因果图。
- **白盒测试**：基于程序逻辑结构。
  - 方法：语句覆盖、判定覆盖、条件覆盖、路径覆盖。

## **第6章 软件维护与演化**

### 1️⃣ 核心知识点

#### （一）软件维护的定义与类型

- 定义：为保持或恢复软件运行性能，对其修改、改进的过程。
- 类型：
  - 改正性维护（纠错）；
  - 适应性维护（应对环境变化）；
  - 完善性维护（改进功能）；
  - 预防性维护（提升未来可维护性）。

#### （二）维护的特征

- 维护成本高（约占生命周期成本的60%以上）；
- 与硬件维护不同，软件维护是逻辑性的持续改进；
- 文档和配置管理是维护成功的关键。

#### （三）再工程与逆向工程

- **逆向工程**：从已有代码推导系统设计与文档。
- **再工程**：在逆向分析基础上重新设计与构造系统。

#### （四）软件演化

- 软件生命周期的自然延续；
- Lehman 定律：软件必须不断演化，否则会逐渐失效。

------

### 2️⃣ 前置依赖与关联

- 承接测试阶段成果；
- 输出：维护日志、版本更新说明、变更管理记录。

- 能分析维护成本的构成；
- 能区分再工程与逆向工程的关系。

## **第7章 软件项目管理**

### 1️⃣ 核心知识点

#### （一）项目管理目标

- 在 **成本、时间、质量** 三者之间取得平衡；
- 任务包括：计划、组织、控制、评估。

#### （二）项目计划

- 内容：范围说明、任务分解、进度安排、资源配置；
- 工具：PERT图、甘特图、WBS（工作分解结构）。

#### （三）成本与进度估算

- 模型：
  - 专家估算法；
  - **COCOMO模型**：通过估算代码行数预测工作量；
  - 功能点分析法（Function Point）。

#### （四）风险管理

- 风险识别 → 分析 → 应对计划；
- 风险类型：技术、进度、成本、人力、需求变更。

#### （五）配置管理（SCM）

- 控制软件变更的过程；
- 工具：Git、SVN、Jira；
- 四要素：识别、控制、审计、报告。

------

### 2️⃣ 前置依赖与关联

- 贯穿软件生命周期；
- 与每个阶段文档管理、版本控制密切相关。

------

### 3️⃣ 学习目标

- 能制定基本的软件项目计划；
- 能使用工具绘制项目进度图；
- 能描述软件配置管理的目标与流程。

------

### 4️⃣ 推荐学习路径

1. 学习项目计划与COCOMO模型；
2. 使用Excel或Project编制进度表；
3. 实践：用Git管理一个小型软件项目。

------

### 5️⃣ 达成标准与自测

- **掌握标准**：能独立编写项目计划书并进行风险分析。
- **自测题：**
  1. 说明配置管理的主要活动及其作用。
  2. 若项目延期10%，请分析可能的原因与改进措施。

------

### 6️⃣ 推荐资源

- Sommerville 第22章《项目管理》
- B站：“COCOMO模型教学”“Git配置管理入门”

## **第8章 软件质量保证与过程改进**

### 1️⃣ 核心知识点

#### （一）软件质量定义

- ISO/IEC 9126质量模型：
   功能性、可靠性、易用性、效率、可维护性、可移植性。

#### （二）质量保证（SQA）

- 活动：评审、测试、标准化、度量；
- 工具：代码审查、同行评审、自动化检测。

#### （三）软件度量

- 过程度量：缺陷密度、平均修复时间；
- 产品度量：代码复杂度（McCabe）、行数、模块数量；
- 项目度量：开发进度偏差、成本偏差。

#### （四）过程改进

- **CMMI模型**：五个成熟度等级
   （初始 → 可重复 → 已定义 → 已管理 → 优化）
- **ISO9001**：强调质量管理体系；
- 持续改进循环（PDCA：Plan–Do–Check–Act）。

------

### 2️⃣ 前置依赖与关联

- 与测试阶段及项目管理相互支撑；
- 输出：质量报告、改进计划、过程评估文档。

------

### 3️⃣ 学习目标

- 能说明软件质量的主要特性；
- 能理解质量保证与质量控制的区别；
- 能解释CMMI模型的基本思想。

------

### 4️⃣ 推荐学习路径

1. 阅读质量模型与CMMI分级；
2. 案例学习：如何建立质量管理体系；
3. 设计简单的过程改进计划。

------

### 5️⃣ 达成标准与自测

- **掌握标准：** 能根据项目质量问题提出改进措施。
- **自测题：**
  1. 解释CMMI的五个成熟度等级。
  2. 如何通过度量指标评估软件质量？

------

### 6️⃣ 推荐资源

- Sommerville 第24章《软件质量管理》
- B站：“CMMI模型入门”“ISO9001质量体系简介”
