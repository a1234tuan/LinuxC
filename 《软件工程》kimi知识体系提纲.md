# 《软件工程》知识体系提纲  
（基于上传 8 份 PPT 的系统梳理与教学化提炼）

---

## 一、课程总体结构

| 模块              | 逻辑地位与作用                                               |
| ----------------- | ------------------------------------------------------------ |
| 1. 软件工程概论   | 建立“工程化思维”，理解软件生命周期与质量目标，为后续所有决策提供价值坐标。 |
| 2. 软件过程模型   | 选择“宏观开发路线图”，决定项目节奏、交付形态与风险管理策略。 |
| 3. 需求工程       | 决定“做什么”，是设计、测试、验收的唯一基准，错误需求=错误系统。 |
| 4. 结构化分析     | 传统数据驱动的需求建模方法，与结构化设计无缝衔接，仍是大量 legacy 系统维护的必备技能。 |
| 5. 结构化设计     | 将需求转化为“模块+接口+数据”蓝图，强调高内聚低耦合与变换/事务映射。 |
| 6. 面向对象分析   | 以对象与职责视角重新建模需求，输出领域模型与用例模型，为 OO 设计奠基。 |
| 7. UML 与建模     | 统一可视化语言，承载需求、设计、实现三阶段沟通，降低认知负荷。 |
| 8. 设计模式与架构 | 提供可复用的“中层+高层”解决方案，平衡质量属性（性能、扩展、安全）。 |
| 9. 测试与质量保障 | 验证“做对”且“做对的事”，贯穿单元→集成→系统→验收四级。        |
| 10. 维护与演化    | 软件 70% 成本所在，涵盖纠错、适应、完善、预防四类活动与回归测试策略。 |
| 11. 项目管理      | 保证“按时、按质、不超支”，覆盖估算、进度、风险、团队与敏捷治理。 |

⚠️ 上传 PPT 对“测试”“维护”“项目管理”着墨较少，建议补充：  
- 测试技术（黑盒/白盒/灰盒、等价类、边界值、路径覆盖、自动化框架）  
- 维护度量（MTTF、MTTR、技术债务）  
- 敏捷估算（故事点、燃尽图、Scrum 角色）

## 二、章节级知识提炼

### 第1章 软件工程概论

#### 1️⃣ 核心知识点
- 软件危机  
  └─ 表现：延期、超支、质量差、不可维护  
  └─ 根源：需求模糊、规模复杂、缺乏规范  
- 软件工程定义（IEEE：系统化、规范化、可量化）  
- 三大目标：低成本、按时交付、高质量（功能/可靠/易用/效率/可维护/可移植）  
- 生命周期：定义 → 开发 → 运维 → 退役  
- 方法学三维：过程 + 方法 + 工具  

#### 2️⃣ 前置依赖
无，为课程入口。

#### 3️⃣ 学习目标
- 能用自己的话解释“软件=程序+数据+文档”  
- 能对比软件与硬件在“制造/维护”上的差异  

#### 4️⃣ 学习路径
阅读 PPT → 观看“软件危机”纪录片 10min → 小组讨论：身边项目失败的根因

#### 5️⃣ 达成标准 & 自测
标准：列举 4 个软件危机现象并给出工程化对策  
自测：  
Q1 为什么“软件不会磨损”却需要维护？  
Q2 给出两条软件质量属性并说明度量方式。

---

### 第2章 软件过程模型

#### 1️⃣ 核心知识点
- 瀑布：阶段顺序、文档驱动、基线评审  
- V-Model：测试级别与开发阶段对应  
- 增量/演化：分批交付、降低风险  
- 螺旋：循环+风险分析+原型，适合大型复杂  
- RUP：二维（阶段×工作流）+ 迭代增量 + 9 核心工作流  
- 敏捷：价值观+12 原则+Scrum/XP 实践  

#### 2️⃣ 前置依赖
第1章生命周期概念

#### 3️⃣ 学习目标
- 能根据项目特征（需求稳定度、团队规模、风险）选择合适模型  
- 能绘制螺旋模型四象限并给出各象限输出

#### 4️⃣ 学习路径
理论 20min → 案例对比（政府办公系统 vs 互联网小程序）→ 课堂投票“选模型”→ 教师点评

#### 5️⃣ 达成标准 & 自测
标准：给定场景（需求常变、团队 5 人、2 个月交付），写出过程模型名称 + 三条理由  
自测：  
Q 瀑布与螺旋的最大差异是什么？  
任务：用 1 张 A4 海报对比 5 种模型适用场景。

---

### 第3章 需求工程（含结构化分析）

#### 1️⃣ 核心知识点
- 需求三层：业务(B) → 用户(U) → 功能(F)  
- 需求工程七活动：起始→获取→细化→协商→规格→确认→管理  
- 获取技术：访谈、问卷、观察、原型、文档考古  
- 结构化分析  
  ├─ 功能建模：数据流图 DFD（上下文→0层→N层）  
  ├─ 数据建模：ER 图（实体、属性、关系 1:1/1:N/M:N）  
  ├─ 行为建模：状态转换图（初态/终态/事件/守卫/动作）  
- 需求规格说明书 SRS（IEEE 830 模板：引言、总体描述、具体需求）

#### 2️⃣ 前置依赖
过程模型（决定需求活动位置）

#### 3️⃣ 学习目标
- 能绘制三层 DFD 并平衡父图子图  
- 能把现实问题抽象为 ER 图并标记主键  
- 能写出符合 SMART 原则的单条功能需求

#### 4️⃣ 学习路径
阅读 → 视频“如何跑需求访谈” → 实验 1：绘制银行储蓄系统 2 层 DFD → 实验 2：将 DFD 转换为 ER → 课堂评审

#### 5️⃣ 达成标准 & 自测
标准：给定“图书自助借还系统”场景，输出：  
- 上下文 DFD 1 张  
- 0 层 DFD 1 张  
- 功能需求 3 条（含前置/后置条件）  
自测：  
Q 验证(Validation) vs 确认(Verification) 区别？  
任务：用决策表描述“借书逾期处理”规则。

### 第4章 结构化设计

#### 1️⃣ 核心知识点
- 设计原则：抽象、模块化、信息隐藏、高内聚低耦合  
- 内聚类型（7 级）：功能→顺序→通信→过程→时间→逻辑→偶然  
- 耦合类型（7 级）：非直接→数据→标记→控制→外部→公共→内容  
- 变换型 vs 事务型数据流 → 映射为系统结构图（SC）  
- 接口设计：人机交互黄金三规则（控制/减负/一致）  
- 数据设计：文件（顺序/索引/哈希） vs 关系数据库（范式→反范式）  

#### 2️⃣ 前置依赖
第3章 DFD 输出

#### 3️⃣ 学习目标
- 能将变换型 DFD 映射为初始 SC 并优化（扇入/扇出、作用域/控制域）  
- 能判断模块内聚等级并提出改进方案

#### 4️⃣ 学习路径
理论 → 例题“银行储蓄”映射 → 小组优化比赛 → 工具 Visio 绘制 SC → 教师打分

#### 5️⃣ 达成标准 & 自测
标准：给定 SC，指出违反的 2 条启发式规则并修正  
自测：  
Q 为什么“公共耦合”会降低可维护性？  
任务：将 3 层 DFD 转换为 SC，再按优化原则迭代 2 次。

---

### 第5章 UML 与面向对象建模

#### 1️⃣ 核心知识点
- UML 三要素：事物（结构/行为/分组/注释）→ 关系（依赖/关联/泛化/实现）→ 图（14 种）  
- 用例图：参与者→用例→关系（include/extend/generalization）  
- 类图：类=名称+属性+操作；可见性（-/+/~/#）；关系：关联→聚合→组合→泛化→实现  
- 交互图：顺序图（时间顺序） vs 通信图（空间链接）  
- 状态图：状态、事件、转换、动作、嵌套/并发状态  
- 活动图：动作节点、控制流、泳道、分叉/汇合  

#### 2️⃣ 前置依赖
第3章需求；第6章OOA概念

#### 3️⃣ 学习目标
- 能识别用例并绘制用例图，正确使用 include/extend  
- 能识别类之间“组合/聚合/继承”差异并正确绘制类图  
- 能把场景转换为顺序图（对象-消息-返回）

#### 4️⃣ 学习路径
MOOC 视频 30min → 工具 StarUML 实战 → 课堂即时测验（5 图 20 分钟）

#### 5️⃣ 达成标准 & 自测
标准：给定“在线点餐”描述，输出用例图+核心类图+下单顺序图  
自测：  
Q 组合与聚合的 2 个区别？  
任务：用 StarUML 导出 PDF 并上传 GitHub 做同行评审。

---

### 第6章 面向对象分析（OOA）

#### 1️⃣ 核心知识点
- OOA 任务：找类→定职责→画交互→抽状态  
- 用例驱动：业务用例→系统用例→用例实现  
- 三类分析类：实体类（持久）、边界类（交互）、控制类（协调）  
- 对象模型 5 层：主题→类-对象→结构→属性→服务  
- 动态模型：顺序图/通信图/状态图互补验证  

#### 2️⃣ 前置依赖
第5章 UML 图基础

#### 3️⃣ 学习目标
- 能从用例文本识别分析类并建立类图  
- 能为具有明显状态变化的类绘制状态图

#### 4️⃣ 学习路径
案例“选课系统”→ 分角色访谈 → 建立用例模型 → 识别分析类 → 评审

#### 5️⃣ 达成标准 & 自测
标准：输出“选课”用例的用例图+三类分析类图+CourseTask 状态图  
自测：  
Q 为何 OOA 阶段先不关注设计模式？  
任务：对比结构化 DFD 与 OO 用例图的 2 个异同。

---

### 第7章 软件体系结构与设计模式

#### 1️⃣ 核心知识点
- 架构=构件+连接件+约束  
- 风格：数据流（管道-过滤器）、调用/返回（分层、MVC）、仓库（黑板）、独立构件（代理）  
- 分布式结构：C/S→三层→B/S→分布式对象（CORBA/RMI/.NET Remoting）  
- 设计模式 4 要素：名称-问题-解决方案-效果  
- 5 大原则：SOLID + 迪米特 + 组合复用  
- GoF 23：创建型 5→结构型 7→行为型 11  

#### 2️⃣ 前置依赖
第5/6章 OO 建模

#### 3️⃣ 学习目标
- 能为给定质量场景（高并发、可扩展）选择合适架构风格  
- 能手写单例、工厂、观察者、策略模式代码并指出开闭原则体现

#### 4️⃣ 学习路径
理论 → 编码 Kata（15min 写单例线程安全版）→ 架构评审演练（ATAM 轻量级）

#### 5️⃣ 达成标准 & 自测
标准：给出“电商秒杀”场景，写出 1 条架构决策+1 个应用模式+理由  
自测：  
Q 为何说“组合优于继承”？用代码片段说明。  
任务：用工厂+策略重构 if-else 计价逻辑并提交 Git。

### 第8章 软件测试与质量（⚠️ 建议补充）

#### 1️⃣ 核心知识点
- 测试级别：单元→集成→系统→验收  
- 测试方法：黑盒（等价类、边界值、决策表、状态转换） vs 白盒（语句/分支/条件/路径覆盖）  
- 回归测试、冒烟测试、探索式测试  
- 缺陷度量：密度、去除率、MTTD、MTTR  
- 自动化金字塔：UI→Service→Unit  

#### 2️⃣ 前置依赖
第3-7章 需求与设计模型

#### 3️⃣ 学习目标
- 能设计等价类测试用例并达到语句覆盖 100%  
- 能编写 JUnit 5 参数化测试

#### 4️⃣ 学习路径
JUnit 入门 Lab → 引入 Mockito 模拟 → CI 中跑测试 → SonarQube 看覆盖率

#### 5️⃣ 达成标准 & 自测
标准：为“订单计算”模块写 5 条 JUnit 用例，覆盖正常/边界/异常  
自测：  
Q 为何 100% 覆盖≠无缺陷？  
任务：将测试报告上传 GitHub Actions 并展示徽章。

### 第9章 软件维护与演化（⚠️ 建议补充）

#### 1️⃣ 核心知识点
- 4 类维护：纠错、适应、完善、预防  
- 再工程：逆向→重构→正向  
- 技术债务：本金+利息，追踪工具（Sonar、CodeScene）  
- 回归测试策略：测试库选择、优先级排序、自动化  

#### 2️⃣ 前置依赖
测试、设计模式

#### 3️⃣ 学习目标
- 能制定“老旧系统迁移”维护计划（含风险评估）  
- 能用 IDE 重构工具完成“提炼方法”并保证测试通过

#### 4️⃣ 学习路径
维护案例“COBOL→Java”→ 识别坏味道 → 重构 → 记录债务

#### 5️⃣ 达成标准 & 自测
标准：在 500 行遗产代码中完成 3 项重构并提交 diff 报告  
自测：  
Q 预防性维护与重构的关系？  
任务：用 Sonar 生成技术债务报告并解释前 3 条债务。

### 第10章 软件项目管理（⚠️ 建议补充）

#### 1️⃣ 核心知识点
- 估算：功能点、COCOMO II、故事点  
- 进度：WBS、关键路径、甘特图、燃尽图  
- 风险：识别→分析→计划→监控  
- 敏捷治理：Scrum 角色、工件、事件；看板 WIP 限制  
- 配置管理：版本控制（Git Flow）、持续集成/交付（CI/CD）

#### 2️⃣ 前置依赖
所有技术章节（估算需需求，CI/CD 需测试）

#### 3️⃣ 学习目标
- 能使用 Planning Poker 完成 1 轮团队估算  
- 能搭建 GitHub Actions 实现提交→编译→测试→部署流水线

#### 4️⃣ 学习路径
理论 → 工具实操 → 模拟 Sprint（4 天）→ 回顾会议

#### 5️⃣ 达成标准 & 自测
标准：输出 Sprint 计划会议视频 + 燃尽图 + 回顾报告  
自测：  
Q 解释“完成定义 DoD”在估算中的作用  
任务：用 GitHub Projects 管理用户故事并自动流转状态。

## 四、使用说明与后续建议

1. 教学顺序：按 1→2→3→6→5→7→4→8→9→10 进行，先 OO 思维再结构化，便于对比。  
2. 实践主线：以“在线选课系统”为贯穿案例，每章输出对应工件，期末整合为完整项目仓库。  
3. 能力检验：采用“章节微测验 + 阶段项目评审 + 期末综合演示”三级制，所有工件纳入 Git 仓库并运行 CI。  
4. 资源索引：  
   - B 站关键词：软件工程 敏捷 生命周期 设计模式 UML  
   - 教材：Sommerville 第 9 版对应章节  
   - 工具：StarUML / PlantUML + JUnit5 + GitHub Actions + SonarCloud
